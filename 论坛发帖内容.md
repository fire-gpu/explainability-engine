## 正文

### 1. 摘要

用 TRAE SOLO 从一个想法出发，经过需求澄清、架构设计、方案选型，到完整实现了一个 **8000+ 行代码的 Python 可解释性分析引擎**。支持文件上传自动分析、因果推断、预测模拟，生成分层报告，并附带 Streamlit 在线 Demo。整个过程一气呵成，SOLO 负责了从 brainstorming 到 coding 的全流程。

### 2. 背景

我是一名 AI 应用开发者，在实际项目中经常遇到一个问题：**模型输出缺乏可解释性**。比如定价策略模型给出了推荐价格，但业务方会问"为什么是这个价格？""如果成本涨 10% 会怎样？"——这些问题很难快速回答。

传统的做法是数据科学家手动跑分析、写报告，耗时且不可复用。我希望做一个**通用的可解释性中间件**，输入模型输出文件，自动生成解释报告，让业务决策者能直接看懂。

### 3. 实践过程

#### 第一步：需求澄清与架构设计（Brainstorming）

我首先调用了 SOLO 的 brainstorming skill，SOLO 通过多轮对话帮我理清了需求：

- **模型无关**：不绑定特定模型，支持任意数据分析场景
- **三层解释能力**：描述性（发生了什么）→ 因果性（为什么）→ 预测性（如果...会怎样）
- **分层报告**：执行摘要面向业务、详细分析面向技术、技术附录面向数据科学家
- **技术栈**：Python + 状态机 + LLM 混合引擎

SOLO 提出了 3 种架构方案并给出了推荐：

| 方案 | 描述 | 评价 |
|------|------|------|
| A: 纯规则引擎 | 状态机 + 硬编码规则 | 可控但因果推理能力弱 |
| **B: 混合引擎 ✅** | **统计分析 + LLM 解释 + 状态机** | **最佳平衡** |
| C: 纯 LLM Agent | LLM 自主完成所有分析 | 不可控、不可审计 |

最终选定方案 B，SOLO 生成了完整的设计文档（含架构图、状态机设计、数据结构、模块划分）。

#### 第二步：任务拆解与并行实现

SOLO 将项目拆解为 6 个阶段，并通过并行 agent 加速实现：

1. **项目脚手架** — pyproject.toml、22 个 dataclass、状态机、编排器框架
2. **基础模块** — 文件解析器、LLM 客户端、规则引擎（15 条预定义规则）
3. **分析引擎** — 描述性分析器、因果推断引擎、预测模拟器
4. **报告生成** — 报告生成器、Markdown/HTML 渲染器、编排器完整实现
5. **测试与配置** — 25 个测试用例、4 个领域 YAML 配置、CLI 入口
6. **发布准备** — README、LICENSE、Git 初始化

每个阶段 SOLO 都自动创建完整可运行的代码，包含中文注释和 docstring。

#### 第三步：添加在线 Demo

为了方便演示，我让 SOLO 额外创建了一个 Streamlit Web Demo（620 行），支持：
- 浏览器端上传文件或使用内置示例数据
- 侧边栏配置领域、目标变量、分析深度
- 四个 Tab 页展示报告
- 一键下载 Markdown/HTML 报告

#### 踩过的坑

- **GitHub 推送认证**：GitHub 已不支持密码认证，需要用 Personal Access Token
- **URL 反引号问题**：从聊天框复制命令时带入了反引号，导致 git remote URL 错误
- **Python 版本**：项目要求 Python 3.11+，macOS 自带版本可能不够，需要用 Homebrew 升级

### 5. 效果与总结

**提效显著：** 从想法到完整项目（8000+ 行代码 + 25 个测试 + 在线 Demo），如果纯手工开发预计需要 1-2 周，使用 SOLO 在一个会话内完成。

**SOLO 在流程中做了什么：**

1. **Brainstorming** — 通过多轮对话帮我理清需求、选型架构、输出设计文档
2. **任务拆解** — 将复杂项目拆解为 6 个可并行执行的阶段
3. **并行编码** — 同时启动多个 agent 实现不同模块，大幅加速
4. **质量保障** — 自动编写 25 个测试用例并全部通过
5. **发布准备** — 自动生成 README、LICENSE、Git 初始化

**可复用的方法：**

- 先 brainstorming 再 coding 的工作流非常高效，避免返工
- 并行 agent 是加速大型项目的关键策略
- 设计文档 → 分阶段实现 → 测试验证 → 发布，这个流程可以复用到任何项目

### 6. 不足与后续改进方向

作为一个 v0.1 版本，当时识别了以下不足：

**🔴 核心短板：LLM 缺乏业务上下文**

当前 LLM 只拿到统计数据和模板 prompt，对业务场景一无所知，生成的解释偏"统计描述"而非"业务洞察"。比如解释"价格变量重要性最高"，但不知道这是电商还是 B2B 场景，无法给出有针对性的建议。

**🟡 报告受众分层不够明显**

业务版和技术版的报告内容差异不大，自然语言不够流畅，模板生成的文字偏机械。

改进路线分三个层级：

| 层级 | 方案 | 效果 |
|------|------|------|
| **L1: 数据语义理解** | LLM 先对数据做"预扫描"——读列名、采样行、统计摘要，自动推断业务场景和指标含义 | 解释更贴合数据本身 |
| **L2: 领域知识注入（RAG）** | 接入 RAG，检索行业知识库（定价策略理论、营销方法论等），让解释有据可依 | 解释有行业深度 |
| **L3: 企业知识对接** | 支持上传企业内部文档（历史报告、业务规范），构建私有知识库 | 完全适配企业场景 |

### 7. v0.2 迭代：L1 数据语义理解 + 报告分层优化（2026-04-12）

基于 v0.1 的不足，用 SOLO 进行了一轮快速迭代，聚焦三个问题：

#### 改动 1：新增数据预扫描模块（L1）

创建了 `llm/data_scanner.py`，在正式分析前让 LLM 先"阅读"数据：

```
用户上传数据 + 输入业务问题/场景（可选）
    → LLM 预扫描（列名 + 前100行 + 统计摘要）
    → LLM 推断：这是什么业务场景？关键指标是什么？可能的因果假设？
    → 将推断结果注入后续所有分析阶段的 prompt
```

用户现在可以通过 CLI 或 Demo 输入：
- `--question`：关键业务问题（如"为什么利润在下降？"）
- `--context`：业务场景描述（如"B2B SaaS 定价分析"）

即使不输入，LLM 也会根据数据自动推断业务场景。

#### 改动 2：报告真正区分业务版和技术版

重构了报告生成器，按受众走完全不同的生成路径：

| 维度 | 业务版 (executive/business) | 技术版 (technical) |
|------|---------------------------|-------------------|
| 语言风格 | 咨询顾问给老板的 brief | 数据科学家的技术报告 |
| 技术术语 | 禁止出现 p值、置信区间、IQR 等 | 完整保留 |
| 因果描述 | "成本上涨推动了价格提升" | "cost -> price: ATE=0.85, p=0.003" |
| 预测描述 | "如果成本降10%，利润预计提升约15%" | 数值 + 置信区间 |
| 结构 | 回答问题 → 3条发现 → 3条建议 | 完整分析 + 方法说明 + 假设条件 |

#### 改动 3：自然语言模板重写

重写了无 LLM 模式下的模板回退，从机械的列表罗列改为流畅的叙述性语言。示例：

> 本次分析围绕定价策略场景下的 9 个变量展开，数据共 50 行。
>
> 核心发现：
> • price 是影响 margin 的最关键因素，其变化直接驱动了利润空间的波动。
> • cost 与 price 之间存在密切关联，提示成本结构对定价策略有显著影响。
> • 数据中检测到部分异常定价记录，可能反映促销活动或定价策略调整。

#### 改动文件清单

| 文件 | 改动 |
|------|------|
| `llm/data_scanner.py` | **新建** — 数据预扫描器 |
| `core/models.py` | 新增 BusinessUnderstanding、AnalysisConfig 新字段 |
| `llm/prompts.py` | 全部 prompt 重写，支持业务上下文注入 |
| `report/generator.py` | 按受众分三路生成，模板重写 |
| `core/orchestrator.py` | 集成预扫描步骤 |
| `analyzers/*.py` | 三个分析器传入 business_understanding |
| `cli/main.py` | 新增 --question / --context 参数 |
| `streamlit_app.py` | 新增业务问题和场景输入框 |

29 个测试全部通过，向后兼容。

### 8. 剩余改进方向

- **可视化图表**：集成 plotly 生成变量重要性图、相关性热力图、因果图等
- **交互式探索**：支持用户"追问"模式，以及 Demo 中拖动滑块实时 What-if
- **L2 RAG 领域知识注入**：让解释从"数据驱动"升级为"行业知识驱动"
- **因果推断增强**：集成 DoWhy / CausalNex，支持用户手动编辑因果图
- **多数据源 / PDF / PPT**：多文件关联、PDF 导出、PPT 自动生成
